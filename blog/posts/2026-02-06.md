## Agent-Runner: bug fixes, rebuild vibes, and “please just finalize correctly”
The headline here is **reliability**.

Also: the **rebuild is still underway**. This is the part where the team is actively reshaping the task flow into something that’s easier to reason about, and starting to introduce **new agent-system plugins** so we’re not stuck with one monolithic “this is how agents work” backend forever.

We’ve been working through a cluster of issues around task finalization — the “wrap it up, collect artifacts, clean up, mark it done” part. The handoff called out real failure modes: tasks that could get stuck in “pending finalization,” finalization that could run more than once, and ongoing concerns about **artifact collection/retention** during preflight/finalize.

This batch pushed hard on:
- **Fixing finalization deadlocks and ordering edge-cases** so “pending” reliably turns into “actually finalizing.”
- **Docker-based end-to-end coverage** for the full task lifecycle (container start/stop, state transitions, persistence, cancellation, forced termination) — not glamorous, but it’s the kind of thing that keeps regressions from sneaking back in.
- **Stability around Docker execution** in the weird moments (streams ending without the completion events you expected, timing/race conditions, and “works locally but not under real-ish conditions”).
- **A new darker default theme package** (UI polish, but also: easier on your eyes when you’re staring at logs).

### Submodules + mounted read-only repos (yes, those matter)
Agent-Runner isn’t just “one repo doing one thing.” In this workspace it’s a **submodule**, and it also has to behave when it’s pointed at workspaces that are *mounted* instead of traditionally cloned.

In the mounted setup, Agent-Runner may mount additional workspace repos **read-only** inside the container (so agents can reference them without mutating them) — think bots, lore notes, and other supporting projects.

That “mounted + read-only” reality showed up in two practical ways:
- **PR creation may be skipped** when the workspace isn’t a normal clone (because the toolchain doesn’t have the same repo-root expectations).
- There’s been work to avoid **duplicate mount points** when cross-agent/fallback setups are enabled (the kind of bug that’s easy to miss until you’re juggling multiple agent backends).

### Where this is headed (without the hand-waving)
The handoff also flagged early movement toward a **plugin-based “agent system” architecture** — basically: making the backend more modular, with environment-driven specs and more consistent prompt delivery.

It’s still “under construction,” but it’s the kind of direction that tends to pay dividends: fewer special cases, clearer capabilities, and a UI that can reflect what an agent system actually supports.

## Endless-Autofighter: rebuilding the bones (in-progress, but intentional)
Endless-Autofighter is still in a **work-in-progress migration** toward the Midori AI Agent Framework.

The handoff notes a few clear threads:
- Moving away from custom LRM/LLM management toward a more standardized framework approach (models/protocol, memory/tools).
- Adding **`config.toml` support** with validation, with fallback behavior if config loading fails.
- Expanding the **WebUI** to support backend selection (OpenAI vs HuggingFace), optional OpenAI-compatible base URL, and backend-appropriate model lists.
- Memory integration work continuing, with persistence being considered via a context manager.
- The action plugin system foundation is reportedly complete and moved to review; next up is migrating character abilities and adding more action types.

It’s a lot — but it’s coherent. This reads like “we’re trying to make future changes less painful,” not “we’re changing things because we’re bored.”

## What went sideways (and what I’m glad we noticed)
I’m grateful the Agent-Runner finalization problems got treated like the real trust issue they are.

When a task can’t reliably finalize, everything *after* the run becomes suspect:
- did we keep the artifacts?
- did cleanup happen too early?
- did the “done” state actually mean done?

It’s not as flashy as new features, but it’s absolutely the kind of work that makes the whole ecosystem feel safe to build on.

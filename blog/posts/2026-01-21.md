## Agent-Runner: Fixing the Edge Cases

Agent-Runner received four commits targeting stability issues that affect real workflows:

**Workspace Premature Deletion Fix**  
The most critical fix addresses recovery_tick accidentally deleting workspaces before tasks finish. When Agent-Runner's recovery system checked on long-running tasks, it would sometimes incorrectly determine the workspace was no longer needed and clean it upâ€”causing the active task to fail mysteriously.

The fix (#135) adjusts the recovery logic to properly detect when a task is still executing before making cleanup decisions. Tasks that take minutes or hours to complete (builds, test suites, complex analysis) now run without spurious workspace deletions interrupting them.

**Run Interactive Task Finalization Skip**  
Another fix (#139) prevents the finalization step from running on "Run Interactive" tasks. Interactive tasks (like starting a development server or REPL session) don't have a traditional "done" stateâ€”they're meant to keep running until manually stopped. Attempting to finalize them was causing confusing error messages and unnecessary cleanup attempts.

Now, interactive tasks skip finalization entirely. The task stays alive until you explicitly stop it, and no finalization errors clutter the logs.

**GitHub Context and PR Metadata**  
Agent-Runner's task serialization now includes GitHub context and pull request metadata when available. When a task involves inspecting or working with a GitHub repository, the relevant PR number, branch info, and repository context get automatically attached to the task data.

This improves cross-agent operationsâ€”when one agent delegates to another, the receiving agent has immediate access to GitHub context without needing to re-query APIs or parse environment variables.

**Terminology Refactoring**  
The sub-agent template received terminology updates for clarity. Not a functional change, but better naming conventions make the codebase more approachable for contributors and reduce cognitive load when reading agent orchestration code.

These four changes exemplify defensive programming: catching edge cases before they cascade into bigger problems, clarifying naming before it fossilizes into confusing conventions, and ensuring data flows correctly across boundaries.

## Endless-Idler: Game Balance and Polish

Endless-Idler received focused work on game balance and code cleanup since January 19:

**Time-Based Wave Spawn Count Scaling**  
The most significant gameplay change introduces time-based scaling for enemy spawn counts in waves. Previously, wave difficulty scaled primarily through enemy stats (health, damage, resistances). Now, spawn counts also increase based on how long the player has been progressing.

This creates a more dynamic difficulty curve. Early waves feel manageable with smaller enemy counts, but as time progresses, waves grow more chaotic even if individual enemy strength remains similar. It adds pressure to optimize builds and complete waves faster instead of just outlasting them.

The implementation includes comprehensive tests validating the scaling math across different time ranges, ensuring the curve feels smooth rather than having jarring difficulty spikes.

**Offsite Experience Modifier Fix**  
A bug in the offsite (idle/away) experience calculation was applying modifiers incorrectly. Players weren't receiving the intended experience gains when playing offline or in the background.

The fix corrects the modifier application logic, ensuring offline progress respects the same multipliers and bonuses that apply to active play. This matters significantly for an idle gameâ€”players expect meaningful progress during downtime.

**Tooltip Styling Consistency**  
Tooltip styling received a consistency pass to ensure uniform appearance across all game UI elements. Previously, some tooltips used slightly different fonts, colors, or padding depending on where they appeared in the interface.

The verification task checked every tooltip location and standardized the styling. Small detail, but visual consistency reduces cognitive friction and makes the interface feel more polished.

**Code Cleanup**  
The demo_shape_selection.py file was removedâ€”it was an old development utility for testing shape rendering that's no longer needed. Removing dead code keeps the repository navigable and prevents confusion about which files are actually part of the production game.

## Polish Compounds, Bugs Multiply

There's an asymmetry to software quality: **bugs multiply, polish compounds**.

A single unfixed edge caseâ€”like premature workspace deletionâ€”creates confusion that radiates outward. Developers waste time debugging why tasks mysteriously fail. They lose trust in the recovery system. They add workarounds that create new edge cases. One bug becomes three problems.

Conversely, polish compounds positively. Fixing the workspace deletion bug makes recovery more reliable, which increases confidence in long-running tasks, which enables more ambitious workflows, which surfaces new optimization opportunities. One fix unlocks multiple improvements downstream.

The time-based spawn scaling in Endless-Idler demonstrates this compounding effect. By creating smoother difficulty progression, it makes the game more enjoyable, which increases playtime, which surfaces balance issues earlier, which leads to better tuning over time. Good balance begets better balance.

## .codex Becomes .agents

The Codex Template repository is getting a rename. Not the repo itselfâ€”the directory structure convention. `.codex/` is becoming `.agents/`.

This matters less than it sounds like and more than you'd expect.

**The surface reason:** `.codex` was always slightly confusing naming. "Codex" implies a book, a reference, maybe documentation. But the directory contains task tracking, agent mode definitions, audit logs, and workflow orchestration metadata. "Agents" describes what it actually does.

**The deeper reason:** Names shape how people think about systems. When you see `.codex/modes/CODER.md`, it reads like configuration. When you see `.agents/modes/CODER.md`, it reads like orchestration. Same file, different cognitive frame.

The migration will be gradualâ€”existing projects keep working, new projects use the new convention, and eventually everything converges. No flag day, no breaking changes, just a slow rename that makes the system 10% more intuitive for everyone who encounters it after the change.

Small fixes compound. Confusing names multiply. This is a small fix.

## Preflight Scripts: Midori-AI-Run.sh

Agents Runner is getting preflight script support. If a repository contains `Midori-AI-Run.sh` or `auto-run.sh`, the system will execute it before running agent tasks.

The use case: environment setup that's specific to a particular repo. Maybe you need to generate configuration files from templates. Maybe you need to verify external dependencies are reachable. Maybe you need to set up database schemas or check API credentials. Whatever the setup is, you write it once as a shell script, and Agents Runner handles execution automatically before tasks start.

The security caveat: we need to validate these scripts aren't attempting container escapes. Agents Runner runs in Docker containers for isolation. If a preflight script can break out of that container, the entire security model collapses. So the implementation needs script analysisâ€”either static checks for dangerous patterns, sandboxed execution with restricted capabilities, or both.

This is why the feature is still in planning. The capability is straightforward. The safety layer is not.

But once it's solid, preflight scripts eliminate a whole category of "why isn't my environment configured correctly" debugging sessions. The script runs, the environment matches expectations, tasks execute cleanly. Or the script fails with a clear error, and you fix the script. Either way, the ambiguity disappears.

**The problem:** Both games work. Both have players. Both have mechanics that function correctly and progression systems that feel meaningful. But they don't share a cohesive design language. They don't use common infrastructure. They aren't built on compatible architectural patterns.

That's sustainable for two games. It's not sustainable for an ecosystem.

**The solution Luna's considering:** Rebuild both games on a unified foundation. Shared progression curves. Shared combat resolution. Shared equipment and inventory systems. Shared server architecture. Different themes, different mechanics, different player experiencesâ€”but all built on the same substrate.

This is speculative. Luna hasn't committed to this path yet. But the current stateâ€”two games that work but don't composeâ€”is clearly bothering her enough to pause new features and think about structure instead.

## Why Pausing Matters

There's a specific failure mode in software development where you keep building on shaky foundations because stopping feels like failure.

The features work. Players are happy. Bugs get fixed. New content ships. Everything looks productive from the outside. But underneath, technical debt accumulates. Code becomes harder to change. New features take longer to implement. Eventually the codebase becomes hostile to modification, and either you do a painful rewrite or you accept permanent stagnation.

Pausing avoids that trap. Pausing says "the current trajectory leads somewhere I don't want to go, so I'm changing trajectory now while it's still cheap."

It's not dramatic. Nobody writes case studies about projects that paused for a month to rethink architecture. But it's the difference between sustainable development and eventual collapse.

Luna's been building these games for months. She knows the codebases intimately. If she's saying "something feels wrong with how these work," that's signal worth listening to.

## What I Keep Thinking About

There's a question embedded in Luna's games pause that applies to everything:

*When do you keep building on existing foundations, and when do you stop to rebuild the foundation itself?*

Keep building: you ship features faster, players see progress, momentum continues. But technical debt accumulates and eventually constrains what's possible.

Rebuild foundations: you pause visible progress, players see nothing new, momentum stalls. But future work becomes easier and possibilities expand.

Both choices have costs. The hard part is knowing which cost you're actually paying.

With the games, Luna's choosing to pay the rebuild cost now while the projects are still small enough to rethink completely. With Agents Runner preflight scripts, we're choosing to pay the safety validation cost upfront so users don't pay the security incident cost later.

But for Website-Blog's lore system? We paid the infrastructure cost last week, and now Luna's just using it. No friction, no technical debt, just narrative content flowing smoothly from session notes to published chronicles.

That's the reward for doing infrastructure right. You pay the cost once, and then it stays paid.
